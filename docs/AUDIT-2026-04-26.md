### Performance wins (C)

- **Revenue + fees N+1 collapsed.** Were: 1 escrow API call per order × 2 periods (≈1000 round-trips for 500 orders). Now: 1 batch call per 50 orders × 2 periods (≈20 round-trips). Same shape via shared `fetchEscrowBatch` helper. Should eliminate the 60s Vercel timeout on the Revenue section.
- **Customers routes**: 3 Supabase queries (orders + states + count) collapsed into 1 paginated rowset query in both Shopee and TikTok routes. TikTok route now uses the FULL state alias list (was using a 5-entry subset).
- **Reconciliation `/fetch` route parallelized** — chunk fetch loop and escrow batch loop both Promise.all'd.
- **`Dashboard.tsx`** got a 60s tick interval so "Updated Nm ago" updates without user action.

### Bug fixes during refactor

1. **🔥 `app/api/reconciliation/fetch/route.ts` was completely broken.** Both `fetchOrderSns` and `fetchEscrowBatch` typed responses as `{ response: { ... } }` and read `res.response?....`, but `callShopee` already strips that wrapper at `lib/shopee.ts:115`. Both helpers silently returned `[]` on every call. **Fixed** during C4.
2. **TikTok new-customer pagination bug** — old query had no `range()`, so Supabase capped at 1000 historical rows and could mis-classify high-volume buyers as "new". Now uses `paginateAll`.
3. **TikTok `STATUS_MAP` inconsistency** — `app/api/tiktok/orders/route.ts` was missing `SELLER_PART_SEND_GOODS`; route showed raw status string while DB stored `SHIPPED`. Now both go through `mapStatus()` from `lib/tiktok-orders.ts`.
4. **Inconsistent error handling** — `app/api/tiktok/customers/route.ts` always returned 500; now branches on `auth`/`rate_limit` like every other route.
5. **Inline prev-period revenue formula** in shopee/revenue/route.ts replaced with `aggregateRevenue(prev)` so current and prev share the same code path.

## Common gotchas while editing this codebase

1. **`callShopee` strips the `response` wrapper.** `(data.response ?? data) as T` at `lib/shopee.ts:115`. Type your response WITHOUT the wrapper. The reconciliation route's old bug came from getting this wrong twice.
2. **MYT date math is everywhere.** All Unix timestamps from Shopee/TikTok are UTC seconds; daily buckets and "this month" comparisons go through `lib/datetime.ts` helpers. Don't reinvent.
3. **Two buyer identity fields.** Shopee uses `buyer_user_id` (BIGINT), TikTok uses `tiktok_buyer_uid` (string). The `orders` table holds both; customer routes filter by `platform`.
4. **Address masking.** Shopee returns `"****"` for masked addresses on MY orders. `unmasked()` in `db-sync.ts` and `normaliseState()` in `states.ts` both filter those out.
5. **`data_source` field on orders.** `'api'` (cron sync), `'report'` (XLSX upload), or `'both'`. The `_upsertOrderRows` function in `db-sync.ts` carefully preserves fields each source can't provide (API path keeps report addresses; report path keeps API buyer_user_ids).
6. **Tests use mocked Redis + Supabase.** See `__tests__/lib/kv.test.ts` and the api tests for the patterns.
7. **`.next/dev/types/` and `.next/types/` cache route validators.** Delete after route deletions or you get phantom TS errors. The handoff workflow at the top of this doc takes care of it.
8. **The user uses Windows + Git Bash.** Path style mostly matters for shell commands. The `Glob`/`Grep`/`Read` tools handle either.
